# 项目案例模板

**项目名称：** 渐进放松助眠游戏  
**完成日期：** 2026-04-27  
**负责方向：** 前端开发（音频系统架构）

---

## 一、项目概述

一个基于Web的渐进放松助眠游戏，通过呼吸引导、身体部位放松和真人语音引导，帮助用户入睡。

**核心功能：**
- 三阶段呼吸引导（短时/长时模式）
- 7种环境音（雨声、海浪、溪流、篝火、虫鸣、白噪音、无）
- 18段真人语音引导（讯飞智作生成）
- 睡眠日程记录

**音频系统架构演进：** 本项目在音频部分经历了重大重构，从最初的`<audio>`元素方案逐步演化为基于Web Audio API的完整解决方案，解决了seek失败、语音-字幕不同步、语音重叠等关键问题。

## 二、架构决策记录

| # | 决策点 | 可选方案 | 最终选择 | 当初假设 | 实际结果 | 假设成立？ |
|---|--------|----------|----------|----------|----------|------------|
| 1 | 语音播放技术 | `<audio>`元素 / AudioBuffer | **AudioBuffer** | 两者都能播放音频，AudioBuffer更底层可控 | AudioBuffer可精确控制播放位置、解决seek问题，且支持事件回调 | ✅ |
| 2 | 语音文件组织 | 单文件+时间戳切片 / 独立文件 | **独立文件** | 独立文件更清晰，但担心文件数量多 | 18个独立MP3文件，需管理映射关系，但避免了同步问题，代码更清晰 | ✅ |
| 3 | 语音状态管理 | 全局变量 / 互斥锁单例 | **VoicePlayer单例+互斥锁** | 需要保证同时只有一段语音播放 | 成功避免语音重叠，stop()能可靠中断当前播放 | ✅ |
| 4 | 环境音生成 | 静态音频文件 / Web Audio API合成 | **Web Audio API合成** | 合成方案更灵活、可编程控制 | 成功生成7种真实环境音，文件体积小，可动态调节参数 | ✅ |
| 5 | AudioContext初始化时机 | init()阶段预加载 / startRelaxation()时加载 | **startRelaxation()时加载** | AudioContext需要用户交互才能创建 | 成功避免19个autoplay警告，用户体验流畅 | ✅ |
| 6 | ambientGain初始化 | initAudio()中创建 / 独立初始化 | **确保initAudio()中创建** | 提前创建audioContext不会影响后续逻辑 | 修复后发现initAudio()中有early return导致ambientGain从未创建，Bug修复 | ❌ |
| 7 | 语音时长获取 | 估算 / 实际获取 | **getDuration()获取** | 估算时长可能不准 | getDuration()在缓存后准确获取，为第一步延迟提供精确数据 | ✅ |
| 8 | 呼吸词处理 | 文字+语音 / 仅文字 | **仅文字** | 呼吸词很短，播放语音可能与步骤引导冲突 | 决策正确，避免了两套音频系统打架 | ✅ |

## 三、踩坑汇总

| # | 现象 | 原因 | 解决方案 | 教训 | 严重程度 |
|---|------|------|----------|------|----------|
| 1 | 语音播放后seek到指定位置失败，声音从头开始 | `<audio>.currentTime`在音频未完全加载时不生效 | 改用AudioBuffer方案，用`source.start(0, offset)`精确控制 | 音频播放的技术选型要一开始就选择底层方案，避免后期重构 | P1 |
| 2 | 语音播放时字幕显示不同步，语音和字幕时间差1-2秒 | 使用`setTimeout(估算时长)`来触发字幕切换 | 改用`source.onended`事件回调，语音播完才切换字幕 | 音频切换绝不能用估算时间，必须用事件驱动 | P1 |
| 3 | 多段语音连续播放时，下一段开始后上一段还在播，产生重叠 | 没有互斥机制，多个音频源同时存在 | 实现VoicePlayer单例+互斥锁，stop()时递增`_playGeneration`代数 | 任何状态管理系统都要考虑并发和中断场景 | P1 |
| 4 | 页面加载时出现19个AudioContext警告："autoplay was blocked" | 在init()阶段创建AudioContext并尝试播放 | 将AudioContext创建移到用户点击"开始"后的startRelaxation()中 | 用户交互前的任何音频操作都可能失败，要延迟到交互后 | P2 |
| 5 | 环境音切换后崩溃："Cannot read property 'connect' of null" | VoicePlayer._ensureContext()提前创建了audioContext，导致initAudio()中检测到已存在后直接return，ambientGain未创建 | initAudio()不再依赖audioContext存在性判断，确保ambientGain始终被创建 | 初始化函数中的early return要检查所有依赖是否真的初始化了 | P0 |
| 6 | 切换步骤时倒计时不准确，切换时机不对 | 倒计时用固定秒数，与呼吸周期不同步 | 倒计时改为breathCycles（呼吸周期数），归零后等当前呼吸周期完成再切换 | 时间相关的逻辑要考虑多个时间源的同步问题 | P1 |
| 7 | 第一步的延迟时间不准确，导致准备页显示时间不对 | 用估算的2秒作为延迟 | 使用`VoicePlayer.getDuration('opening')`获取实际时长，未缓存时默认2.5秒 | 任何外部资源的时长都不要硬编码，要实际获取 | P2 |
| 8 | Gitee Pages无法开启，提示需要实名认证 | Gitee政策要求实名认证才能使用Pages功能 | 改用ZIP文件分发作为临时方案 | 平台功能限制要提前调研，避免后期卡住 | P2 |

## 四、如果重做 反思

### 1. 当初的选择：使用讯飞智作生成真人语音
**实际暴露的问题：** 
- 讯飞智作需要手动操作：编写脚本→上传→选择音色→生成→下载
- 18段语音需要逐个生成和管理
- 语音与代码耦合，需要维护voiceFiles映射

**重做会怎么做：**
- 考虑使用Web Speech API（TTS）直接在前端合成语音
- 或使用讯飞TTS API实现自动化生成
- 将语音文件存放在独立目录，通过配置文件管理映射关系

### 2. 当初的选择：使用GitHub Pages作为部署方案
**实际暴露的问题：**
- GitHub在国内访问不稳定，用户无法打开
- 需要翻墙或国内镜像

**重做会怎么做：**
- 一开始就选择Netlify拖拽部署（无需账号）
- 或选择Vercel（国内访问较好）+ Gitee Pages（实名认证后）
- 提前评估目标用户的网络环境

### 3. 当初的选择：使用`<audio>`元素播放语音
**实际暴露的问题：**
- currentTime seek不可靠
- 无法精确控制播放位置
- 事件机制不够灵活

**重做会怎么做：**
- 一开始就使用Web Audio API + AudioBuffer方案
- 虽然学习成本更高，但避免了后期大规模重构
- AudioBuffer更适合需要精确控制的场景（游戏音效、语音导航）

### 4. 当初的选择：呼吸动画用CSS @keyframes驱动
**实际暴露的问题：**
- 呼吸圈动画和倒计时是两个独立的时间源
- 切换步骤时，动画和倒计时不同步

**重做会怎么做：**
- 所有时间相关的动画都用requestAnimationFrame驱动
- 建立统一的时间源，所有动画和倒计时从同一个时间基准计算
- 避免CSS动画和JS逻辑混用导致的时间不一致

### 5. 当初的选择：环境音使用预录音频文件
**实际暴露的问题：**
- 需要维护多个音频文件
- 音效参数（音量、频率）固定，难以动态调节

**重做会怎么做：**
- 继续使用Web Audio API合成（这是正确的选择）
- 但要考虑Web Audio API的兼容性问题，准备fallback方案
- 考虑将合成逻辑封装成独立模块，便于在不同项目中复用

## 七、音频制作专项总结

### 学到了什么

1. **Web Audio API比`<audio>`元素更强大**
   - 可精确控制播放位置
   - 支持事件驱动
   - 可动态生成音效
   - 适合游戏和需要精确控制的场景

2. **状态管理的重要性**
   - 互斥锁机制避免音频重叠
   - 代数（`_playGeneration`）防止异步回调在状态切换后执行
   - 单例模式确保全局唯一状态

3. **事件驱动优于时间驱动**
   - 用`onended`事件而不是`setTimeout`估算
   - 用`requestAnimationFrame`同步动画
   - 用Web Audio API的时间而不是JS Date

4. **合成音效的技巧**
   - 棕噪音（Brown Noise）= 白噪音积分
   - 粉噪音（Pink Noise）= 特定频率权重
   - 啁啾声 = 高频叠加周期性envelope
   - 环境音 = 噪音底流 + 特征脉冲 + LFO调制

### 走了哪些弯路

1. **从`<audio>`到AudioBuffer的重构**（浪费约2小时）
   - 应该在项目初期就选定AudioBuffer方案

2. **语音时长估算问题**（浪费约1小时）
   - 应该一开始就实现`getDuration()`缓存机制

3. **AudioContext初始化时机**（浪费约1小时）
   - 应该在设计阶段就考虑autoplay限制

4. **环境音与语音的音量平衡**（持续调优）
   - 合成环境音和录制语音的音量需要分别调节

### 下次如何改进

1. **音频方案选择流程：**
   - 评估是否需要精确控制（游戏/语音导航）→ 是 → Web Audio API
   - 评估是否需要动态生成（环境音/特效） → 是 → Web Audio API合成
   - 简单背景音乐 → `<audio>`元素即可

2. **预研清单：**
   - autoplay政策影响
   - 移动端音频限制
   - 降级方案（iOS Safari Web Audio兼容）

3. **代码规范：**
   - 所有音频操作封装在AudioManager类中
   - 禁止直接操作AudioContext
   - 统一使用事件驱动而非setTimeout
